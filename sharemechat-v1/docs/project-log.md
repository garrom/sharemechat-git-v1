# Bitácora del proyecto SharemeChat

Registro cronológico inverso (más reciente arriba) de hitos relevantes del proyecto: decisiones estructurales cerradas, cierres de fase o paquete, hipótesis archivadas, cambios de rumbo y aprendizajes operativos significativos. Cada entrada captura **qué pasó, por qué y, cuando aplica, qué se descartó y por qué**.

Este fichero **lo mantiene el agente local** (Claude Code u otro agente con acceso al repo) cada vez que materializa un cambio que cae en las categorías definidas. NUNCA se edita a mano. NO se rellena retroactivamente. NO se reescriben entradas pasadas.

La política operativa completa (categorías que disparan entrada, formato fijo, reglas de redacción y de inviolabilidad) vive en `docs/documentation-governance.md`, sección **Política operativa de la bitácora**.

Deudas registradas para futuras tandas (no abordadas en este cierre):
- **P1 (a cerrar antes del go-live PROD)**: la idempotencia en `processDiditWebhook` no distingue "ya procesado correctamente" de "rechazado por firma". Hoy el lookup `findByProviderAndProviderEventId(PROVIDER_DIDIT, event_id)` bloquea ambos casos. Si en PROD el secret se rota mal o se mal configura, los webhooks rechazados quedan como zombies en BD impidiendo reintentos del mismo event_id. - **P2 (sin urgencia)**: aceptar también `X-Signature-V2` por si Didit deprecase Standard en algún momento. Requiere canonicalización JSON estable en Java (Jackson con `SerializationFeature.ORDER_MAP_ENTRIES_BY_KEYS` recursivo, `JsonInclude.Include.ALWAYS`, ascii-no-escape, formato float consistente) y testing cuidadoso de payloads con floats y unicode. Hoy Didit envía las tres variantes en cada request, así que validar Standard es funcionalmente equivalente.
- **P3 menor**: actualizar el `state-mapping.yaml` local de TEST de "arranque manual" a "systemd con `sharemechat-test.service`" — el frente Didit reveló que TEST ya tiene unit systemd creada después del refactor PRO-9, no documentada en el mapping de la IA. Coherente con AUDIT/PROD.
- **P4 menor**: añadir tests unitarios para `KycProviderConfigService` ahora que maneja tres modos (`VERIFF`, `MANUAL`, `DIDIT`) sin red de seguridad — el frente Didit amplió `MODE_DIDIT` y `isDiditEnabledForModelOnboarding()` sin que existiera test previo para `setModelOnboardingMode`.

Estado final de entornos: **TEST** con `kyc.didit.enabled=true` en runtime, workflow conectado al sandbox Didit, `users.id=87` APPROVED como prueba viva, BD prístina, PID 34206 vivo desde `23:38:41 UTC` del 2026-06-13. Próximos frentes en cola (no comprometidos en este cierre): cliente Didit del flujo de CLIENTE (Adaptive Age Verification con Age Estimation primaria y step-up documental, workflow distinto al del modelo, decisión arquitectónica pendiente sobre si reusa `model_kyc_sessions` o necesita tabla nueva `client_kyc_sessions`); activación de Didit en AUDIT (replica de pasos 2+4 con credenciales propias del entorno); frente frontend de "verificación enviada/aprobada/rechazada" tras KYC modelo (hoy no hay UI Didit en producto); cierre obligatorio de la deuda P1 antes del go-live PROD del 1 de julio de 2026.

## 2026-06-13 — Frente Didit paso 2: esqueleto del cliente para el flujo modelo, modo MOCK + replay protection nativa

Implementación del esqueleto backend del cliente Didit para el flujo KYC del modelo (Document + Selfie + Liveness vía Workflow Builder), siguiendo ADR-035. Vendor único Plan A. Se replica el patrón Veriff (commits `27796bb`..`53c3036`) en métodos paralelos sin refactor a strategies — decisión arquitectónica para no acoplar el frente todavía: Veriff queda dormido pero integrado como contingencia técnica reactivable, así que conviene mantener ambos paths claros y aislados; refactorizar a una capa abstracta tiene sentido sólo cuando Veriff se elimine del repo (que no es decisión de este paso).

**Ficheros nuevos:** [`config/DiditProperties.java`](../src/main/java/com/sharemechat/config/DiditProperties.java), [`dto/DiditCreateSessionResult.java`](../src/main/java/com/sharemechat/dto/DiditCreateSessionResult.java), [`service/DiditClient.java`](../src/main/java/com/sharemechat/service/DiditClient.java), [`service/DiditClientImpl.java`](../src/main/java/com/sharemechat/service/DiditClientImpl.java). **Tests nuevos:** `DiditClientImplTest` (10 tests: MOCK + payload + validación de campos opcionales con la misma lección que el incidente Veriff 400/1104 del 2026-06-11) y `ModelKycSessionServiceDiditTest` (18 tests: mapeo case-sensitive de los 10 estados Didit + replay protection con la ventana de 300s + extractores de `event_id`/`webhook_type`/`session_id`/`status`). **Ficheros extendidos:** `ModelKycSessionService` (constructor de 9 args ahora, métodos `startDiditSession` + `processDiditWebhook` con replay protection ANTES de la firma + idempotencia por `event_id` explícito + mapeo a `Constants.VerificationStatuses`); `KycProviderController` (endpoints `/api/kyc/didit/{start,webhook}` con headers `X-Signature` y `X-Timestamp`); `SecurityConfig` (matchers explícitos `hasRole("USER")` y `permitAll`, sin wildcard `/api/kyc/**` para no abrir permitAll por descuido); `ProductOperationalModeService.isAlwaysAllowed` (whitelist exacta para el webhook); `Constants.KycModes.DIDIT`; `KycProviderConfigService.MODE_DIDIT` + `isDiditEnabledForModelOnboarding()` (se SUMA a VERIFF/MANUAL, no sustituye); `application.properties` (bloque `kyc.didit.*` con `enabled=false`); `application-{test,audit,prod}.properties` (cada uno con su `kyc.didit.callback-url`).

**Verificado contra la doc oficial Didit (`docs.didit.me`, 2026-06-13)** — tres divergencias respecto al plan original a destacar:
- Endpoint real: `POST https://verification.didit.me/v3/session/` (no v1, no v2).
- Header de autenticación: `x-api-key` (no `Authorization: Bearer`). El JavaDoc de `DiditProperties.apiKey` lo deja claro para que nadie lo confunda con el patrón Veriff (que sí firma el body de salida).
- Campo de respuesta: `url` (no `verification_url`). El JSON de respuesta es plano, no anidado bajo `verification` como en Veriff.

**Divergencias funcionales clave que el código materializa** (no son sólo notas en doc — son lógica nueva en `processDiditWebhook`):
- **Replay protection nativa**: header `X-Timestamp` con ventana de 300s validada ANTES de la verificación HMAC. Veriff no tiene esto. Si el timestamp falta, no es numérico o cae fuera de la ventana, se persiste fila en `kyc_webhook_events` con `is_signature_valid=false` + `processing_error_message="invalid_timestamp"` y el controller responde 401.
- **Idempotencia con `event_id` explícito**: Didit lo envía en el payload, no hay que derivarlo de `attemptId` como en Veriff. Más limpio.
- **Status case-sensitive directo**: 10 estados (`Approved`, `Declined`, `In Review`, `In Progress`, `Not Started`, `Abandoned`, `Expired`, `Kyc Expired` con K mayúscula, `Resubmitted`, `Awaiting User`). El test `caseSensitive_lowercaseDoesNotMatch` blinda que somos estrictos: `"approved"` minúscula NO matchea `"Approved"` y cae a PENDING (no se asume APPROVED por defecto).
- **No hay HMAC saliente**: el cliente sólo manda el header `x-api-key`. El `secret_shared_key` (mapeado a `kyc.didit.api-secret`) se usa SÓLO para verificar webhooks entrantes (variante "Standard" de Didit: HMAC-SHA256 hex sobre el raw body, header `X-Signature`, reutilizando 1:1 el helper `HmacSha256.verifyHexHmacSha256` ya validado en Veriff).

**Modo MOCK preservado**: cuando `kyc.didit.enabled=false` (estado actual en los tres entornos) o el `api-key` está vacío, el cliente devuelve un `sessionId="didit_mock_<UUID>"` sin llamar a la API real. Mismo patrón que Veriff. Permite arrancar TEST/dev sin credenciales y sin coste accidental.

**Estado de entornos: NO se ha activado nada.** `kyc.didit.enabled=false` en `application.properties` y sin override en `application-{test,audit,prod}.properties`. Las únicas keys per-environment son las tres `callback-url`. AUDIT y PROD no se han tocado. El cliente MOCK funciona; las credenciales reales (api-key + secret_shared_key) y el `workflow_id` llegarán como acción documentada en ADR-035 §"Acciones inmediatas" tras solicitar DPA a `hello@didit.me` y configurar el workflow en consola.

**Tests:** 131 verdes, 0 fallos, 0 errores (`mvn test`). Sin regresión en `ModelKycSessionServiceMappingTest` (21 tests Veriff) ni en `VeriffClientImplTest` (8 tests). El constructor extendido de 9 argumentos en `ModelKycSessionService` obligó a actualizar la fixture `svc()` del test Veriff a 9 nulls — la nota en JavaDoc del test deja el rastro de por qué.

## 2026-06-13 — ADR-035: consolidación del vendor de KYC en Didit (supersede ADR-029)

Veriff Support confirmó por escrito que **Age Estimation no está en el plan Essential** (~49 USD/mes, self-serve) sino en el plan **Enterprise** (~199 USD/mes, contrato anual, mínimo 1000 verifs/mes), inviable para una OÜ pre-ingresos con runway de 12+ meses sin facturación. Tras investigación de vendors alternativos y revisión de los requisitos comerciales y regulatorios (Segpay *Adult Content Due Diligence Checklist*, card schemes, Ofcom UK, US post-Paxton), se cambia el proveedor — NO el método. ADR-029 se mantiene íntegro en cuanto a método y disparadores; cambia sólo quién lo ejecuta.

Vendor único: **Didit** para los dos flujos. Modelo: Didit KYC bundle (ID Verification + Passive Liveness + Face Match + Device & IP Analysis), gratis hasta 500/mes, después 0,33 USD/bundle. Cliente: workflow "Adaptive Age Verification" en consola Didit con Age Estimation facial primaria (0,10 USD/check) y step-up automático a Document+Selfie IDV en buffer zone (challenge age provisional 25), todo orquestado dentro del propio workflow de Didit; el backend recibe un único webhook con el resultado final, sin orquestar el fallback. Datos en UE (AWS Ireland) por defecto. Webhooks HMAC-SHA256 reutilizan el helper `com.sharemechat.security.HmacSha256` ya validado en el frente Veriff. Detalle completo + planes alternativos B/C/D en [ADR-035](06-decisions/adr-035-age-and-identity-verification-vendor-consolidation-on-didit.md).

**Frente Veriff dormido, no eliminado**: los commits `27796bb`..`53c3036` quedan integrados en el repo con `kyc.veriff.enabled=false` en `application.properties` y en `config.env` de los tres entornos. Reactivar Veriff es un flag + credenciales reales, no una reintegración técnica. Esto sostiene los planes alternativos C (Veriff modelo + Didit cliente) y D (vuelta total a Veriff) sin coste técnico permanente.

**Acciones inmediatas registradas** en el ADR-035 (no ejecutadas): escribir a Segpay confirmando aceptación del método con Didit; solicitar DPA + TOMs a Didit (`hello@didit.me`, no hay portal público); diseñar el workflow Adaptive en consola Didit; configurar retención a 6 meses (no dejar el default "unlimited"); iterar DPIA con Didit como processor (entregable que estaba ya en `compliance-deliverables.md` §6.1); implementar el cliente Didit en backend reutilizando el patrón Veriff (HMAC, country gating, idempotencia, mapeo por código de decisión, `byte[]` raw body, `permitAll` + whitelist permanente del webhook).

Notas documentales: ADR-029 marcado al final con status update "SUPERSEDED by ADR-035". `known-debt.md` no se toca (las deudas técnicas del frente Veriff están cerradas). `psp-strategy.md` y `compliance-deliverables.md` siguen referenciando ADR-029 en varios sitios; no se actualizan en esta tanda — la lectura de ADR-029 lleva al ADR-035 vía su status update. Se actualizarán cuando esos documentos se editen por otro motivo. AUDIT y PROD NO tocados; sin cambios de código.

## 2026-06-13 — VEREDICTO en la cabecera del email del `prod-access-reporter`

Cierre del frente abierto al detectar que el email diario de PROD no llevaba la cabecera VEREDICTO pese a que el classifier (commit `056b7e9`) ya la generaba en `.table.txt` y `.summary.jsonl`. El responsable era el reporter, que componía su propio resumen a partir de las filas IP y no leía el verdict. Tras importarlo al repo en el commit anterior (`acb4da8`/`6408330`), este frente aplica el cambio funcional: `lib/report_access.py` ahora detecta la línea `{"type":"verdict",…}` que el classifier emite como primera línea del `.summary.jsonl`, la separa del resto de filas en `load_summary_rows` y antepone un bloque al cuerpo del email con emoji (🟢/🟡/🔴), etiqueta (`[V]/[A]/[R]`), nivel (`VERDE/AMARILLO/ROJO`), label, summary y `reasons`/`notes` si los hay, todo dentro de un marco de barras `=`.

**Defensa en profundidad**. Si el verdict no aparece — classifier antiguo, fichero pre-VEREDICTO, parse error de la primera línea — `load_summary_rows` deja `verdict=None`, escribe un warning a stderr (`[report_access] no verdict block found in <day>.summary.jsonl; falling back to legacy header …`) y el renderer cae al formato anterior sin romper el envío. El `render_verdict_block` además va envuelto en `try/except` por si algún campo trae tipo inesperado: log a stderr y formato legacy. El email se manda siempre; lo único que cambia entre con-verdict y sin-verdict es la presencia del bloque.

**Beneficio incidental**. La versión anterior del reporter contaba toda fila del `.summary.jsonl` como "IP" en `len(rows)`. Tras el deploy del classifier nuevo, esa cuenta se desviaba en 1 (la fila verdict): el journal del 13-jun 07:10 mostraba `IPs analizadas: 85` mientras los counts CRITICA/MALICIOSA/SOSPECHOSA/NORMAL sumaban 84. Con el filtrado nuevo, `IPs analizadas` ya cuadra con la suma de counts.

**Smoke validado**. Ejecutado en local contra `/tmp/.../2026-06-12.summary.jsonl` (CON verdict) y `2026-06-11.summary.jsonl` (SIN verdict, día anterior al deploy del classifier). El primero pinta el bloque, el segundo cae al fallback con el warning a stderr. Tras el deploy a `/opt/sharemechat-prod-access-reporter/lib/`, smoke real sobre el día 12 ejecutado dentro del propio EC2 con `report-prod-access.sh --config … --date 2026-06-12` (sin `--send-email`): exit 0, cuerpo idéntico al smoke local, bloque VEREDICTO arriba.

**Deploy**. Backup del fichero anterior preservado como `.bak-20260613-pre-verdict-email` en el mismo directorio del EC2. `__pycache__/` borrado para forzar que el siguiente run no use bytecode viejo. Permisos `0644` igual que el original. No se toca `bin/`, `config/`, ni el `README.md` del componente. La unit systemd (`sharemechat-prod-daily-report.service` + `.timer`) tampoco cambia: el `ExecStart` sigue encadenando classifier + reporter, y la próxima ejecución programada (`domingo 14 jun 07:10 UTC`, sobre eventos del 13 jun) ya saldrá con la nueva cabecera.

**No se replica a AUDIT ni a TEST en este frente**. El classifier del repo solo emite verdict en `prod-access-classifier`; los classifiers de AUDIT (`audit-access-classifier`) y de TEST (`test-access-classifier`) no tienen la lógica. Replicar el reporter ahí ahora mismo no aporta nada (siempre caería al fallback). Cuando se decida portar VEREDICTO a los classifiers de AUDIT/TEST, ese frente arrastrará la replicación de los reporters por coherencia. Anotado como deuda futura en `known-debt.md`. Además, en el momento del deploy AUDIT estaba `stopped` por el scheduler de fin de semana recién activado y TEST estaba inaccesible vía SSH (timeout, deuda preexistente registrada en `ops/deploy-state/test.yaml`); aunque la decisión hubiera sido replicar, el contexto no lo permitía.

**Pieza viva en el repo**. `sharemechat-v1/ops/prod-access-reporter/lib/report_access.py` queda a sha256 `3ebcd245…`, idéntico al fichero servido en `/opt`. El siguiente cambio en cualquiera de los reporters debería usar el mismo patrón aditivo (con fallback explícito a stderr) para que el correo nunca deje de enviarse por una incompatibilidad de schema entre classifier y reporter.

## 2026-06-13 — Cierre del drift del `prod-access-reporter`: import al repo sin cambios funcionales

Drift detectado durante la verificación del bloque VEREDICTO en el email diario de PROD del 2026-06-12: el operador reportó que el correo no traía la cabecera VEREDICTO arriba pese a que el classifier nuevo (commit `056b7e9`) ya estaba commiteado, pusheado y desplegado en `/opt/sharemechat-prod-access-classifier/`. Inspección en `prod-backend` confirmó tres hechos. (1) El classifier nuevo **sí** está corriendo: el `2026-06-12.table.txt` y `2026-06-12.summary.jsonl` locales arrancan con el bloque VEREDICTO esperado (línea 1: `VEREDICTO: [V] SIN ACCION -- 22 IPs MALICIOSA/CRITICA, sin exito en rutas sensibles; 366 eventos totales; nada que revisar.`). (2) La cabecera que el operador ve en el cuerpo del email la compone un **reporter separado** (`sharemechat-prod-access-reporter`, encadenado tras el classifier por la unit `sharemechat-prod-daily-report.service`); ese reporter genera su propio resumen a partir de las filas y solo cita los `.table.txt`/`.summary.jsonl` como fuentes, sin leer el verdict. (3) Y aquí está el drift: **el componente `prod-access-reporter` vivía únicamente en `/opt` del EC2 de PROD, no estaba versionado en el repo**, mientras que sus equivalentes `audit-access-reporter` y `test-access-reporter` sí lo están.

Este commit cierra el drift importando el componente al repo sin cambios funcionales. ## 2026-06-13 — AUDIT weekend scheduler: artefactos creados y validados, schedules en DISABLED hasta OK del operador

Frente nuevo de ahorro operativo: programar el apagado y arranque automáticos del entorno **AUDIT** los fines de semana usando **EventBridge Scheduler**. Objetivo: AUDIT apagado de viernes 22:00 Madrid a lunes 07:20 Madrid; resto de la semana sigue 24/7 como hasta ahora. SOLO AUDIT. TEST y PROD intactos.

**Decisión de mecanismo: EventBridge Scheduler con targets universales `aws-sdk:*`**, no Instance Scheduler (CloudFormation oficial de AWS). Instance Scheduler trae Lambdas, DynamoDB y CloudWatch logs propios — overhead injustificado para 2 recursos y 4 disparos/semana. Targets universales (`arn:aws:scheduler:::aws-sdk:ec2:stopInstances`, etc.) llaman al SDK directamente desde el servicio Scheduler, sin Lambda intermedia. Coste ~0. Timezone `Europe/Madrid` interpretada por Scheduler, cambio CET ↔ CEST automático.

**Orden de operaciones**. Apagado: EC2 primero (viernes 22:00 Madrid, drena conexiones HTTP/WS antes que la BD), RDS después (viernes 22:20 Madrid). Arranque: RDS primero (lunes 07:00 Madrid, BD lista antes que la app), EC2 después (lunes 07:20 Madrid). `sharemechat-audit.service` está `enabled` en el host → systemd lo arranca solo al boot, sin intervención manual.

**Separación de perfiles AWS y mínimo privilegio**. Dos perfiles distintos por filosofía operativa: `sharemechat-provisioner` (control plane, potente, uso acotado y documentado) y `sharemechat-deployer` (default, runtime, mínimo necesario). El frente requería tres operaciones de control plane:
1. Crear policy IAM `sharemechat-scheduler-audit-deployer` y attachearla al user deployer (extender el deployer con scope estrecho para que pueda crear/actualizar los schedules del grupo del frente).
2. Crear rol IAM `sharemechat-scheduler-audit-role` (asumido por `scheduler.amazonaws.com`) con inline policy de start/stop EC2 + RDS sobre ARNs literales.
3. Mover la ventana de backup RDS AUDIT (`rds:ModifyDBInstance`, no permitido al deployer).

Las tres se hicieron con `AWS_PROFILE=sharemechat-provisioner` (uso puntual, no recurrente). El resto (Scheduler create/update/get + lectura RDS/EC2 + smoke tests + futuras ejecuciones del operador) se hace con el deployer y nada más. **No se hardcodea ningún perfil en los scripts del repo** (`apply.sh`, `toggle.sh`); ambos toman el default del entorno.

**Scope mínimo en la policy IAM del deployer**. `iam:PassRole` acotado al ARN del rol del scheduler **y** con condición `iam:PassedToService=scheduler.amazonaws.com` (aunque deployer llegara a nombrar el rol en otro contexto, solo Scheduler podría recibirlo). `scheduler:Create/Update/Delete/Get` acotados a `schedule-group/sharemechat-audit-weekend` y `schedule/sharemechat-audit-weekend/*`. Las únicas dos acciones que requieren `Resource: *` son `scheduler:ListSchedules` y `scheduler:ListScheduleGroups` (la API del servicio Scheduler no admite filtrado IAM por nombre de grupo en las llamadas de listado); justificado en el README del frente.

ARNs literales, no comodines ni tag conditions: `ec2:Start/StopInstances` sobre `instance/i-0d9149cd8a0e24104`; `rds:Start/StopDBInstance` sobre `db:db1-sharemechat-audit`. Motivo: en invierno (CET=UTC+1) el viernes a las 21:20 UTC = 22:20 Madrid se apaga la RDS, **13 minutos antes** del inicio de la ventana de backup actual; el snapshot del viernes no se ejecutaba ese día. Sábado, domingo y lunes 02:00 UTC la RDS sigue apagada (arranca lunes 05:00–06:00 UTC según DST), por lo que esos días no hay snapshot — esperado y coherente con el ahorro. **Lo que NO se tocó**: `BackupRetentionPeriod=1`, `MultiAZ=false`, `DeletionProtection=false`, `PreferredMaintenanceWindow=mon:03:32-mon:04:02`, ni TEST ni PROD.

## 2026-06-12 — Cabecera VEREDICTO operativo en el clasificador de accesos de PROD

Adición puramente aditiva al `prod-access-classifier` (Python, cron systemd cada 15 min, encadenado tras el `normalizer`). El clasificador sigue generando exactamente lo mismo de antes (`.summary.jsonl` con una fila por IP + `.table.txt` con tabla tabular); lo nuevo es un **bloque VEREDICTO** al inicio de cada output que resume si el día requiere o no acción del operador. Pensado para que, de un vistazo en el email perimetral, se sepa si toca abrir el reporte completo o ignorarlo.

**Filosofía VERDE por defecto**. El operador trabaja sobre la base de que en un día normal de PROD todos los sondeos terminan en 404 (el producto no expone superficies hostiles reales) y, aunque la tabla muestre muchas IPs MALICIOSA o CRITICA según el scoring agregado del classifier, eso no es accionable en sí mismo. VERDE no se rompe a menos que haya evidencia concreta: éxito real en una ruta sensible, éxito de auth-abuse, señal de DoS, o un pico estadístico claro respecto a la línea base.

**Niveles y reglas**:
- **ROJO — ACTÚA**: una IP no-allowlisted obtuvo `2xx` (solo `2xx`, no `3xx`) en una ruta sensible (`.env`, `.git`, `.aws`, `actuator`, `wp-admin`, `wp-login`, `xmlrpc.php`, `cgi-bin`, `vendor`, `phpmyadmin`, `config`, etc.); o `2xx` en `/api/auth/login` desde IP con `≥5` fallos previos; o `>100` POST a `/api/auth/*` o `/api/users/register/*` desde una IP; o señal de DoS (`>200k` eventos totales en el día, o `>5000` requests desde una IP).
- **AMARILLO — REVISAR** (raro por diseño): solo dos condiciones — (1) pico de IPs CRITICA `>3×` la media de los últimos 7 días Y `≥15` en absoluto; (2) `2xx` en un endpoint `/api/*` PROTEGIDO (cualquier `/api/*` que no sea `/api/public/*` ni `sitemap.xml`/`robots.txt`) desde una IP no-allowlisted. Esa segunda regla es la única forma fiable de detectar "fuga real" en este sistema: cualquier ruta del frontend devuelve `200 + index.html` por el SPA fallback (`try_files $uri /index.html`), así que un `200` en `/login` o cualquier otra ruta extensionless **NO** es éxito del atacante; solo lo es un `2xx` en un endpoint protegido del API.
- **VERDE — SIN ACCIÓN** en cualquier otro caso. Si una IP no-allowlisted ha sondeado rutas `/api/*` no-pública-conocida y ha recibido siempre 401/404 (`≥10` sondeos), el dato se conserva como **nota informativa** dentro del bloque VERDE para visibilidad, sin elevarlo a AMARILLO (es la autenticación funcionando, no es accionable).

**Decisiones de diseño aprendidas durante el ciclo de prueba sobre datos reales de PROD (06-08 a 06-12)**:
- *Solo `2xx`, no `3xx`, cuenta como éxito real*. La regla inicial aceptaba `2xx/3xx` y el `301` que nginx emitía para `/wp-login.php` (redirect SPA fallback / HTTPS canónica) disparaba un ROJO falso positivo sobre la IP `159.203.62.198` del 06-10. Un `3xx` no implica que el atacante haya leído el recurso protegido.
- *Whitelistar rutas SPA no resuelve el ruido del SPA fallback*. La regla "2xx en ruta no-publica-conocida" se planteó primero con una lista de rutas SPA conocidas (`/login`, `/forgot-password`, `/blog`, etc.); pero un scanner pide rutas aleatorias que también reciben `200` por el fallback y no estarían en esa lista. La regla refinada IGNORA por completo cualquier `2xx` del frontend y restringe la detección a `2xx` en `/api/*` protegido — eso es fuga real y debería ser casi inexistente.
- *El sondeo de `/api/*` real sin éxito NO es accionable*. Una IP que prueba `/api/admin`, `/api/config`, `/api/me`, `/api/auth/me`, etc. y recibe siempre 401/404 está chocándose con la autenticación — no hay nada que el operador deba hacer. Se elimina como motivo de AMARILLO y se conserva como nota informativa dentro del VERDE para visibilidad.
- *El multiplicador del pico y su floor combinados*. El multiplicador 2.5× con floor 5 disparaba AMARILLO en días tranquilos cuando la línea base era 2-3 IPs y subía a 6-8 (no es un pico real, es ruido). Subido a 3× con floor 15: AMARILLO solo cuando hay un salto claro Y al menos 15 IPs CRITICA en absoluto.

**Validación sobre datos reales de PROD (5 días)** ejecutada antes del deploy al cron: 2026-06-08 VERDE, 2026-06-09 AMARILLO (solo por pico real de 21 IPs CRITICA vs media 6.7), 2026-06-10 VERDE, 2026-06-11 VERDE con nota informativa (sondeo de `195.178.110.104` a 10 rutas `/api/*` no-pública-conocida × 115 requests, todos 401), 2026-06-12 VERDE. Distribución: 4 VERDES / 1 AMARILLO / 0 ROJOS, exactamente el objetivo de ruido bajo y AMARILLO/ROJO genuinamente raros.

**Cambios al código**. Solo aditivos: una nueva sección al final de `lib/classify_access.py` (~500 líneas) con la función `compute_verdict`, helpers `read_critical_baseline` y `build_verdict_summary`, render `render_verdict_block`, y constantes `VERDICT_*` con todos los umbrales. La función `run_classifier` se modifica mínimamente para invocar el cálculo del veredicto entre el cálculo de las assessments y la escritura de los outputs; `write_summary` acepta un parámetro opcional `verdict` que escribe como primera línea. La lógica original `[0/5]..[5/5]`... — perdón, la lógica original del classifier (scoring por IP, allowlist short-circuit, ordenado por severidad) queda **intacta**. Cero cambio de comportamiento sobre la tabla y las filas; solo se añade contexto.

**Deploy al cron de PROD**. Backup del fichero anterior conservado en EC2 como `.bak-20260612-pre-verdict-block`. Una ejecución manual de validación tras el deploy confirmó que el `.table.txt` del día actual arranca con la línea VEREDICTO esperada y el `.summary.jsonl` con `"type":"verdict"`. systemd timer y service intactos.

## 2026-06-13 — Veriff paso 6: mapeo del Decision webhook al contrato real de Veriff (verification.code como autoridad)

Fix descubierto en el paso 4-bis (sesión 2026-06-13 forzando "declined" en Station para la sesión id=10): el webhook entrante llegaba con firma válida (`is_signature_valid=1` en `kyc_webhook_events`), pero el mapeo de estados no actualizaba `kyc_status` ni `users.verification_status`. La sesión id=10 quedaba en `provider_status=success`, `kyc_status=PENDING`, `decided_at=NULL`.

**Diagnóstico contra el payload real** (capturado en la fila id=3 de `kyc_webhook_events`). El Decision webhook de Veriff tiene esta estructura: `{ status: "success", verification: { code: 9102, status: "declined", id: "<sessionId>", attemptId: "<attemptId>", reason: "Suspected document tampering", reasonCode: 102, ... } }`. Dos confusiones críticas:

- El código viejo leía el `status` **top-level** ("success" = "webhook recibido OK") y lo interpretaba como decisión; el `verification.status` real ("declined") quedaba ignorado.
- El extractor `extractDecisionCode` devolvía un `String` que iba al matcher de keywords ("declined" / "approved"), pero Veriff envía `verification.code` como **entero numérico** (9001 / 9102 / 9103 / 9104 / 9121 / …), no como string semántica.

Resultado: ningún branch del matcher acertaba; caía al fallback `optString("status")` → leía "success" top-level → `provider_status="success"`, `kyc_status=PENDING`.

**Política de mapeo aplicada** (autoridad: `verification.code` numérico):

| `verification.code` | `kyc_status` interno | Razón |
|---|---|---|
| 9001 | APPROVED | Decisión final aprobada |
| 9102, 9103, 9104 | REJECTED | Decisión final rechazada |
| 9121 | PENDING (resubmission) | Flujo sigue abierto; el usuario debe reintentar |
| otros / null | PENDING + log warn | Nunca asumir APPROVED por defecto |

`provider_status` se persiste **literal** desde `verification.status` (sin interpretar): útil para auditoría. `decided_at` se rellena solo cuando el code mapea a decisión final (APPROVED/REJECTED), nunca para RESUBMISSION ni desconocido. `submitted_at` queda en NULL: NO se deduce del decision webhook (eso es del event webhook, integración futura). `users.verification_status` se actualiza solo en decisión final; para RESUBMISSION/desconocido se preserva el PENDING previo del usuario (el flujo sigue abierto).

**Mini-fix de extractores** (event_id / event_type / decision_code / decision_reason). El payload real no tiene `id`/`eventId`/`event_id` top-level ni `eventType`/`action`/`type`. Se reescriben los extractores para:

- `provider_event_id`: prefiere **`verification.attemptId`** (único por intento, sostiene idempotencia). Cae a top-level `id`/`eventId`/`event_id` por compatibilidad con event webhooks futuros. Si todo falla devuelve null aceptado: la fila se persiste igual y la reconciliación se sostiene por `provider_session_id`.
- `provider_event_type`: usa `eventType`/`action`/`type` top-level si existen; si no, **deriva `"decision_<verification.status>"`** (p. ej. `"decision_declined"`) para distinguir en logs sin inventar contrato.
- `extractDecisionCode` pasa a devolver `Integer`. Lee `verification.code` con `getInt`; fallback a string numérica por defensa.
- `extractDecisionReason` lee `verification.reason` (preferido) o `verification.reasonCode` (numérico, lo stringifica).

**Tests** (`ModelKycSessionServiceMappingTest`, **21 nuevos**): code 9001/9102/9103/9104/9121/desconocido/null mapeados correctamente; `extractDecisionCode` lee int / string-numérica / null; `extractProviderStatus` lee `verification.status` literal **NO** el top-level; `extractProviderEventId` prefiere `attemptId`, cae a top-level, devuelve null aceptado; `extractProviderEventType` deriva `decision_<status>` o usa top-level explícito; `extractDecisionReason` lee `reason` y stringifica `reasonCode`. ## 2026-06-09 — Fix puntual del check de drift para sesiones NO interactivas (nuevo flag `-AssumeYesNonCritical`)

Corrige una limitación detectada en el primer deploy real con el frente integrado (commit `3751b63`): el `Read-Host` del paso `[0.5/N]` requiere consola interactiva. En sesiones sin consola (la IA via harness PowerShell con stdin redirigido, automation en CI, scheduled tasks), `Read-Host` aborta con `"Windows PowerShell se encuentra en modo no interactivo"`. En el primer deploy real esto forzó a usar `-SkipDriftCheck` para llegar al `[5.5/N]`, lo que **anula la protección justo para los deploys que ejecuta la IA, que son los que causaron el incidente del 2026-06-08**. Fix puntual que cierra este agujero sin romper el comportamiento interactivo humano ni la lógica existente `[0/5]..[5/5]`.

**Nuevo parámetro: `-AssumeYesNonCritical`** (switch). Semántica exacta:
- El `[0.5/N]` **SÍ corre y SÍ evalúa la severidad** (no se salta, no como `-SkipDriftCheck`).
- En severidad `OK`/`INFO`: continúa sin prompt (igual que antes).
- En severidad `WARN` o `ALERT`: si la sesión es **no interactiva** + `-AssumeYesNonCritical`, auto-confirma y continúa sin `Read-Host`. Si la sesión es interactiva, se mantiene el `Read-Host [s/N]` original (el flag no cambia nada para humanos).
- En severidad `CRITICAL`: **ABORTA SIEMPRE en no interactivo, con o sin el flag**. Solo un humano interactivo escribiendo `'yes'` literal, o el operador usando `-SkipDriftCheck` conscientemente, puede pasar `CRITICAL`. El flag deliberadamente NO ofrece esa puerta para que la IA nunca pueda auto-saltar el contrato.
- **Sin el flag y en sesión no interactiva**: el `[0.5]` **aborta limpiamente** ante cualquier severidad `>= WARN` (fail-safe). NO cuelga esperando input, NO continúa a ciegas. Mensaje claro al operador indicando que invoque con `-AssumeYesNonCritical` o desde consola humana.

**Detección "host realmente interactivo"**. Helper privado `Test-DeployIsInteractiveHost` en el cuerpo del `[0.5/N]`. Devuelve `$false` si `[Console]::IsInputRedirected` es `$true` (stdin redirigido por pipe, `< file` o harness IA), o si `[Environment]::UserInteractive` es `$false` (servicio, scheduled task, sesión sin escritorio). Devuelve `$true` solo si ambos chequeos pasan. La función se evalúa una vez y se cachea en `$script:IsInteractive`.

**`-SkipDriftCheck` queda como estaba**, sin cambios: salta el `[0.5/N]` entero. Es la vía excepcional para reparaciones conscientes (revert de un deploy roto, primera puesta a punto del manifest). NO es la vía normal para sesiones no interactivas: para eso está `-AssumeYesNonCritical`, que SÍ ejecuta el check y solo auto-confirma WARN/ALERT.

**Validación previa al commit** (modo `-DryRun` en mi sesión, que `[Console]::IsInputRedirected` reporta como `$true`):

- **Caso (a)** ALERT + `-AssumeYesNonCritical`. Manifest real de AUDIT (backend `1cb43a0`, candidato `3751b63`, no toca contrato). Salida: `Severity ALERT` con razones correctas + `[DryRun + non-interactive + -AssumeYesNonCritical] severity ALERT: el deploy real CONTINUARIA (auto-confirmado).` ✓
- **Caso (b)** CRITICAL + `-AssumeYesNonCritical`. Manifest sintético con backend `b54a0be` del 30-may (estado pre-incidente). Salida: `Severity CRITICAL` con los 8 ficheros del contrato listados + `[DryRun + non-interactive] severity CRITICAL: el deploy real ABORTARIA SIEMPRE. (Ni -AssumeYesNonCritical permite continuar; solo humano interactivo escribiendo 'yes' o -SkipDriftCheck consciente.)` ✓
- **Caso (c)** ALERT en no interactivo SIN el flag. Manifest real de AUDIT. Salida: `Severity ALERT` con razones correctas + `[DryRun + non-interactive sin flag] severity ALERT: el deploy real ABORTARIA (fail-safe).` Sin cuelgue, sin continuar a ciegas. ✓

**Comportamiento interactivo humano intacto**. En consola real, `[Console]::IsInputRedirected` es `$false` y `[Environment]::UserInteractive` es `$true`: `Test-DeployIsInteractiveHost` devuelve `$true`, y las tres ramas (`WARN`/`ALERT`/`CRITICAL`) ejecutan el mismo `Read-Host` que antes (`[s/N]` en `WARN`/`ALERT`, `'yes'` literal en `CRITICAL`). Los flags `-Strict`, `-DryRun`, `-SkipDriftCheck`, `-AllowDirtyWorkingTree` mantienen su semántica intacta. La lógica `[0/5]..[5/5]` original no se toca.

**Workflow recomendado por contexto**:
- Operador humano en consola → invoca normal, responde `[s/N]` / `'yes'`.
- IA en sesión non-interactive → invoca con `-AssumeYesNonCritical`; el deploy se aborta automáticamente si hay drift de contrato (CRITICAL); pasa fluido en WARN/ALERT (drift menor sin riesgo de contrato).
- Reparación consciente (revert, primera puesta a punto del manifest) → `-SkipDriftCheck`.

## 2026-06-08 — Hardening del saneado Markdown del blog (Lote 2 parte 1: H4 whitelist de `class` + H5 neutralización del cierre del wrapper `:::callout`)

Segunda tanda de mitigaciones de la auditoría defensiva sobre `MarkdownRendererService` (el motor que convierte Markdown del CMS en HTML servido por `/api/public/content/articles/{slug}` al frontend).

**H4 — whitelist de valores para el atributo `class`**. `addAttributes("div", "class")` de jsoup permitía cualquier valor en `<div>`, `<code>` y `<pre>`. jsoup no soporta whitelist de valores nativamente, así que la defensa se hace post-`Jsoup.clean` re-parseando el HTML limpio con `Jsoup.parseBodyFragment`, iterando `getAllElements()` y filtrando `class` por (tag, valor): `<div>` → solo `callout`; `<code>`/`<pre>` → solo `^language-[A-Za-z0-9_+\-]+$`. El resto se elimina; si no quedan clases útiles, se quita el atributo `class` entero. Sin riesgo de tag stripping: el filtro solo toca el atributo. Cero impacto en artículos actuales (los publicados hoy no llevan callouts ni code blocks con lenguaje en cuerpo).

**H5 — neutralización de `</div>` literal en el cuerpo del `:::callout`**. `preprocessCallouts` envolvía el contenido capturado por regex en `<div class="callout">...</div>`. Si el autor escribía `</div>` literal en el cuerpo, cerraba prematuramente el wrapper (riesgo XSS bajo porque jsoup balancea después, pero el HTML quedaba estructurado raro). Fix: dentro del replacement de la regex, ejecutar una segunda regex case-insensitive `</\s*div\s*>` → `&lt;/div&gt;` sobre el cuerpo antes de generar el wrapper. Implementación con `Matcher.replaceAll(MatchResult → String)` + `Matcher.quoteReplacement` para que `$` y `\` del contenido no se interpreten como backrefs. El `:::` dentro del cuerpo NO requiere tratamiento: el regex de cierre exige `^:::[ \t]*$` (línea entera), un `:::` inline no matchea.

**Decisión de diseño H5**. La alternativa era implementar una extensión flexmark (BlockParser + NodeRenderer + factory + tests del custom Node, ~200 líneas para una sintaxis no estándar). Se descartó por desproporción coste-beneficio: la neutralización en regex son 2 líneas, ataca el único vector real (desbalance de wrapper) y no introduce dependencia ni surface de mantenimiento adicional. Si en el futuro se quiere lookup de callouts dentro de fenced code blocks (caveat v1 documentado), entonces sí compensa la extensión.

**Verificación**. Suite JUnit (`MarkdownRendererServiceTest`) ampliada de 12 a 16 tests, todos pasan: `renderDivWithCalloutClassIsPreserved`, `renderDivWithNonWhitelistedClassLosesClass` (anteriormente documentaba el caveat ahora cerrado), `renderCodeBlockLanguageClassIsPreserved`, `renderCodeBlockBogusClassIsStripped`, `preprocessCalloutWithDivCloseInBodyIsNeutralized`. JAR `d5586f0f...` desplegado a TEST (backup `.bak-lote2-md-20260608-094007`); artículos publicados existentes en TEST (`elegir-videochat-seguro` ES, `how-to-choose-safe-video-chat` EN) renderizan idénticos a antes (cero regresión visual; `htmlBody` 7745 y 7456 chars respectivamente, sin clases en el output porque ningún artículo actual usa callouts).

## 2026-06-08 — Auditoría defensiva: hardening de auth Lote 1 (rate-limit IP en login, validación nickname, escape HTML en plantillas email)

Cierre del primer lote de mitigaciones de la auditoría de seguridad defensiva sobre nuestros propios endpoints. Fase 0 (inventario) confirmó que la única vía de entrada al backend desde IP no-allowlistada es `/api/public/content/**`, `/sitemap.xml`, `/robots.txt`, `/api/users/register/*`, `/api/auth/{login,refresh,logout}`, `/api/email-verification/**`, `/api/consent/**` y webhooks PSP. El **buscador del blog es 100% client-side** (`useMemo` sobre `articles` ya cargados, máximo 50 elementos) → cero superficie de SQLi o XSS reflejado por ese vector. Las queries del CMS público son JPA derived parametrizadas; el `slug` se valida con `^[a-z0-9]+(?:-[a-z0-9]+)*$` antes de tocar BD; el `htmlBody` de los artículos pasa por `MarkdownRendererService` con `Safelist` jsoup explícita sin `<script>`/`<iframe>`/`<svg>`/atributos JS.

Sobre ese inventario, la Fase 1 (inputs públicos e inyección) detectó cuatro hallazgos con severidad media o relevante a corto plazo (más cinco menores aplazados al Lote 2/3). Este lote cierra tres de ellos.

**H3 — rate-limit por IP en `POST /api/auth/login`**. `ApiRateLimitService.checkLoginIp(ip)` existía desde el primer cableado del rate-limit pero **nunca se invocaba**. `AuthController.login` solo llamaba a `checkLoginEmail(dto.getEmail())`. Vector: un atacante con lista de emails reales podía sostener fallos masivos desde una IP estable, pagar solo el coste del lockout-por-email (`AuthRiskService.isEmailBlocked`) y dejar denegado el login a cuentas legítimas, sin coste por IP. Fix: añadir `checkLoginIp(ip)` antes de `checkLoginEmail`, con la IP real procedente de `IpConfig.getClientIp(req)` (no spoofeable tras el perímetro CDN-aware ADR-032: nginx sobrescribe `X-Forwarded-For = $remote_addr`).

**H2 — validación de `nickname` en los DTOs + escape HTML en `EmailCopyRenderer`**. `UserClientRegisterDTO` y `UserModelRegisterDTO` exigen email `@Email` y password `@Size(min=10)` desde V1, pero `nickname` solo era `@NotBlank` (sin tamaño máximo ni patrón). Cualquier payload llegaba a BD y a `User.nickname`. Vector real: `EmailCopyRenderer.renderWelcome/Unsubscribe/Verification/AssetRejection` inyectan `safeLabel(user)` y `displayName` directamente en text blocks HTML vía `.formatted(...)`, 14 sitios distintos; `safeLabel` solo hace `trim()`. Un nickname `<script>alert(1)</script>` llegaba **literal** al cuerpo HTML del email — aunque los clientes de email modernos sandboxean el script, el escape correcto es defensa básica y obligatoria. Fix en dos planos: (a) `@Size(min=3, max=30) + @Pattern("^[\\p{L}\\p{N}._-]{3,30}$")` — letras unicode, dígitos unicode y `. _ -` únicamente; rechaza por construcción los cinco caracteres que rompen estructura HTML (`< > & " '`) más espacios y control chars. (b) helper `htmlEscape(String)` privado en `EmailCopyRenderer` (escapa `& < > " '` a entidades) aplicado a los cuatro orígenes (`nickname` × 2, `displayName` × 2) que alimentan los 14 puntos de inyección. La defensa en profundidad cubre además **nicknames legacy** registrados antes de añadir la validación y el caso de `renderVerification(... String nickname ...)` que recibe el nickname por argumento externo (call-sites: `EmailVerificationService`, retries admin) sin garantía de saneamiento upstream.

**H6 — `limit_req` en nginx, solo en endpoints abusables**. El nginx PROD no tenía `limit_req` global tras ADR-032 (real_ip y override XFF), así que cualquier path quedaba sin throttling de capa 7. Aplicar `limit_req` a todo `/api/` estrangularía crawlers en `/api/public/content/**` (Googlebot pide artículos a ritmo no humano cuando indexa), por lo que el límite se acota a tres prefijos (`/api/auth/`, `/api/users/register/`, `/api/admin/auth/`) vía `location ^~` (prefix preferente, gana al `location /api/` genérico). Zona `auth` con `$binary_remote_addr` keyed por IP real, `rate=30r/m`, `burst=20 nodelay`. Los crawlers y `/api/public/content/**` siguen sin tocar.

**Sinks frontend no requieren cambio**. `AdminAdministrationPanel`, `AdminAssetModerationPanel`, `AdminActiveStreamsPanel`, `AdminDataPanel` y vistas cliente que imprimen `nickname` lo hacen siempre como `{user.nickname}` en JSX — React escapa por defecto. No hay `dangerouslySetInnerHTML` consumiendo `nickname` en todo el codebase frontend.

**Validación en TEST**. Backend recompilado (`f88c1975...`, 106 MB) y desplegado a `sharemechat-test.service`; nginx TEST recargado con backup `bak-h6-20260608-085256` del .conf previo. Smoke: `POST /api/users/register/client nickname="a"×200` → **400**; `POST /api/users/register/client nickname="<script>alert(1)</script>"` → **400**; `POST /api/users/register/client nickname="pepito-test_123"` → 403 (consent/country, **NO** 400 — el pattern pasa); `POST /api/auth/login × 30` desde misma IP → **429** tras agotar burst (nginx emite el 429 antes de tocar Spring; `checkLoginIp` queda como segunda capa); `GET /api/public/content/articles?locale=es × 5` → **200 × 5** (sin throttling); `GET /sitemap.xml × 5` → **200 × 5** (sin throttling). PROD pendiente del mismo deploy (mismo JAR, mismo patrón nginx) — sale en commit aparte al aplicarse.

**Lo que no se cierra en este lote**. Cinco hallazgos pendientes documentados en `docs/04-operations/known-debt.md` como sección nueva "2026-06-08 — Auditoría defensiva: Lote 2 y Lote 3 pendientes". Sumario: **Lote 2** — H1 enumeración de email/nickname en registro (respuesta indistinguible "si los datos son válidos te enviaremos email" + email a la dirección original si ya existe cuenta), H4 atributo `class` libre en jsoup Safelist (`<div class>`, `<code class>`, `<pre class>` actualmente sin whitelist de valores), H5 regex injection del preprocessador `:::callout` (impacto residual = HTML mal balanceado, no XSS; mejor procesar callouts en AST flexmark). **Lote 3** — H8 bumps de dependencia (`jjwt 0.11.5 → 0.12.x`, `jsoup 1.17.2 → 1.18.x`) + ejecutar `mvn org.owasp:dependency-check-maven:check` para inventario CVE actualizado; refactor de la integración CCBill a una capa PSP-agnóstica (`PaymentSession`, `WebhookSignature`) ahora que Segpay queda como candidato.

## 2026-06-05 — Pre-launch industrial (`PRODUCT_ACCESS_MODE=PRELAUNCH`) + dos bugs colaterales arreglados

Se cablea end-to-end el modo coming-soon industrial sobre la capa Product Operational Mode existente (ADR-009). El enum `Mode.PRELAUNCH` y los gates REST/WS ya estaban implementados desde el primer hardening de ADR-009, pero nunca se habían activado en runtime. Esta sesión los activa, expone el estado al frontend, materializa una pantalla pre-launch con la marca, y arregla dos bugs colaterales que el cableado destapó.

**Diseño principal**. El comportamiento se controla por configuración de runtime por entorno, no por bundle: un único build se despliega indistintamente a TEST, AUDIT y PROD; lo que decide es `PRODUCT_ACCESS_MODE` del `config.env` de cada entorno. La reversión real al lanzamiento es `PRODUCT_ACCESS_MODE=OPEN` + restart del servicio, sin rebuild. Backend expone al frontend en `GET /api/users/me` dos campos nuevos (`productAccessMode` ∈ {OPEN, PRELAUNCH, MAINTENANCE, CLOSED} y `allowlisted: boolean`); el frontend monta una puerta única en `RequireRole` que, si el modo es PRELAUNCH y el user no está en `product.access.allowlist.user-ids`, sustituye cualquier ruta de producto por `<PreLaunchScreen/>`. El cierre real de endpoints sensibles (`/api/models/**`, `/api/clients/**`, `/api/transactions/**`, etc.) vive en el gate del backend (`ProductOperationalModeFilter` + `ProductOperationalModeWsInterceptor`), ya existente; la pantalla es solo la experiencia visible. **Cero recogida ni almacenamiento de documentos de identidad para no-allowlisted**: el filter REST devuelve 503 `PRODUCT_UNAVAILABLE` para POST a `/api/models/documents` y `/api/models/kyc`. Aplicado solo a TEST (`PRODUCT_ACCESS_MODE=PRELAUNCH`, `PRODUCT_REGISTRATION_CLIENT_ENABLED=true`, `PRODUCT_REGISTRATION_MODEL_ENABLED=true`, `PRODUCT_ACCESS_ALLOWLIST_USER_IDS=30,31`). AUDIT sigue en OPEN para los PSPs; PROD intacto.

**Bug colateral #1 — backend bloqueaba login/refresh en PRELAUNCH**. El método `ProductOperationalModeService.isProductPath` incluía `/api/auth/login` y `/api/auth/refresh` como "rutas de producto", por lo que en PRELAUNCH devolvían 503 `PRODUCT_UNAVAILABLE` para todos los no-allowlisted. La allowlist no salvaba porque al hacer login todavía no hay cookie con userId, así que el bypass por usuario es imposible. Resultado observable: ningún usuario podía autenticarse en PRELAUNCH; las cookies de sesiones previas además se borraban porque `/api/auth/refresh` con sesión no-backoffice limpia el access_token y refresh_token en el response. **Fix**: en `decideForRequest`, dentro de `case PRELAUNCH`, login y refresh se devuelven siempre `Decision.allow("prelaunch_auth_always_open")` antes de evaluar `isProductPath`. `MAINTENANCE` y `CLOSED` quedan intactos: para esos modos sigue bloqueado login/refresh (semántica original preservada).

**Bug colateral #2 — frontend interpretaba el 503 de PRELAUNCH como mantenimiento**. El interceptor `isMaintenanceResponse` de `config/http.js` (paquete 10.A.3.pre, red de seguridad SPA para ventanas de mantenimiento del backend) consideraba cualquier 502/503/504 como "backend caído" y disparaba `notifyMaintenance(true)`, que monta el overlay full-screen bilingüe hardcoded `MaintenanceOverlay`. En combinación con el bug #1, todo intento de autenticarse activaba la pantalla de mantenimiento — explicación de la sintomatología que el operador veía como "página de mantenimiento desde S3" cuando en realidad era un overlay React montado sobre la SPA. **Fix**: si la respuesta 503 trae header `X-Product-Mode: PRELAUNCH` o `CLOSED`, NO se dispara el overlay (cierre operacional deliberado, no fallo). 503 con `X-Product-Mode: MAINTENANCE` sí lo dispara (semántica alineada con el nombre del modo). 502/504 y 503 sin header (gateway error real) siguen disparándolo.

**Bug colateral #3 — puerta abierta por defecto durante carga de sesión**. React 17 + `ReactDOM.render` (no `createRoot`) no batchea setStates después de un `await`. En `SessionProvider.loadMe()` hay un `setUser(data); await applyLocale(data); setLoading(false);` que produce dos renders distintos: primero `(user=data, loading=true)`, luego `(user=data, loading=false)`. La puerta original `if (loading && !user) return null;` evaluaba el primer render con `user=data` ya populado → no entraba al null, avanzaba por los checks de role y, si `user.productAccessMode` no estaba poblado (cache de navegador con response pre-enriquecimiento, race del effect de `location.pathname` que dispara dos veces durante el mount, etc.), `isPreLaunchActive(user)` devolvía `false` → la puerta se abría → `<ModelDocuments />` (o el dashboard correspondiente) se montaba durante unos milisegundos antes de que el siguiente render reevaluara. Síntoma observable: flicker visible de la página real entre la pantalla blanca y la `PreLaunchScreen`, tanto en hard-refresh como tras cambio de idioma (que hace `window.location.assign` por restricción del basename del Router v5 — recarga completa siempre). **Fix**: política "puerta cerrada por defecto". Mientras `loading=true`, retornar `null`. Mientras `user.productAccessMode` sea `''` / undefined, retornar `null`. Solo cuando ambos están confirmados, decidir `PreLaunchScreen` vs producto real. Coste asumido: un parpadeo "blanco → PreLaunchScreen" de ~100-300 ms en el primer render post-recarga (mientras `/api/users/me` está en vuelo); admisible porque sustituye al flicker grave de "componente real → PreLaunchScreen". Complementado con un merge defensivo en `SessionProvider.updateUiLocale` (`setUser((prev) => ({ ...prev, ...updatedUser }))` en lugar de `setUser` plano) para que el `PUT /users/me/ui-locale`, que devuelve un `UserDTO` sin los campos enriquecidos del GET, no machaque `productAccessMode` y `allowlisted` del estado previo.

**Pase estético de la `PreLaunchScreen`**. La primera versión usaba estilos inline; se reescribió reutilizando los componentes de marca del sitio público: `NavbarBase` + `DesktopActions` + `MobileMenu` (mismo logo y mismo styling responsive que la home), y `HeroContainer` + `HeroOverlay` + `HeroContent` + `HeroCopy` + `HeroTitle` + `HeroSubtitle` para la estructura del hero. Imagen de fondo siguiendo EXACTAMENTE la convención del hero de la home (`${ASSETS_BASE}/<carpeta>/hero/<archivo>_{desktop,mobile}_v1.webp` resuelto en runtime por hostname a `assets.test.sharemechat.com`, `assets.audit.sharemechat.com`, `assets.sharemechat.com`). Se sube a TEST `prelaunch/hero/prelaunch_{desktop,mobile}_v1.webp` en el bucket `assets-sharemechat-test1`. Copy nuevo más corto y con voz de marca (cliente flexionado masculino, modelo flexionado femenino, "founding models" / "modelos fundadoras"). Botón "Cerrar sesión" + botón "Reenviar email de verificación" (este último reutiliza `/api/email-verification/resend`, que está en la whitelist permanente del gate y sigue funcionando en PRELAUNCH). Cero estilos inline.

**Aprendizajes operativos**. (1) Un cierre operacional por configuración (PRELAUNCH) NO debe interpretarse como mantenimiento accidental: la red de seguridad SPA del paquete 10.A.3.pre era demasiado agresiva y necesitaba discriminar por header `X-Product-Mode`. (2) Cuando un endpoint enriquece campos solo en una operación (GET) pero los devuelve también en otra (PUT), todos los call sites del frontend deben usar merge defensivo, no replace; la deuda correcta es enriquecer el backend de forma consistente en TODAS las respuestas del DTO, no parchar cada call site. (3) Una puerta de acceso debe cerrarse por defecto, no abrirse por defecto: ante estado desconocido o sesión a medio cargar, render `null` (o splash) es preferible al render del componente real con corrección posterior. (4) React 17 sin createRoot tiene ventanas de render no-batched después de await que rompen invariantes sutiles; cualquier check basado en multi-state hay que validarlo en runtime.

Lo que NO se cierra en este lote: enriquecer `productAccessMode` y `allowlisted` también en `PUT /users/me/ui-locale` y en cualquier otro endpoint que devuelva `UserDTO` (deuda técnica abierta — el merge defensivo es un parche, la solución correcta es backend consistente); refactor del basename del Router para que el cambio de idioma no requiera `window.location.assign` (deuda mantenibilidad — el reload deja un costoso vacío blanco antes de mostrar la pantalla pre-launch); blindaje del script de deploy de frontend con builds secuenciales y smoke tests obligatorios, requisito antes de PROD (deuda crítica abierta — un incidente menor de la sesión vació temporalmente el bucket producto cuando dos builds compitieron por la carpeta `build/`); sustituir el `return null` de carga por un splash/loader con marca para evitar el blanco breve; subir los assets de hero pre-launch a `assets-sharemechat-audit` y `assets-sharemechat-prod` cuando esos entornos pasen a PRELAUNCH (ahora solo en TEST). Todo recogido en `known-debt.md`. Snapshot del estado TEST tras el cierre en `docs/_snapshots/state-test-2026-06-05-0030.yaml`.

## 2026-05-31 — La navegación interna de producto no preservaba el prefijo /en

**Síntoma reportado por el operador**: un usuario logueado en `/perfil-client` cambia el idioma con el control de la cabecera; la página recarga en `/en/perfil-client` (inglés, correcto), pero al pulsar "Volver" del perfil el idioma vuelve a español porque la URL de destino ya no llevaba el prefijo `/en`.

**Causa raíz**: el basename del Router (`/en` cuando aplica) se fija una vez por carga de página en `App.jsx` a partir de la URL. Las navegaciones que **pasan por React Router** (`<Link>`, `history.push`) heredan ese basename y preservan `/en` automáticamente, pero las que se **saltan el Router** no: `history.goBack()` (nativo, usa la pila del navegador), los `<a href>` absolutos a rutas internas, y `window.location.*` no intencionales. Tras un cambio de idioma (que recarga vía `window.location`), pulsar "Volver" con `history.goBack()` llevaba a la entrada previa del historial, sin `/en` → al recargar, App remonta con basename `/` → español.

**Decisión**: no pelear contra ADR-022. La URL sigue siendo la fuente de verdad del idioma; simplemente se hace que esas navegaciones internas pasen por React Router (`history.push` / `<Link>`), que ya reañade el basename. No se toca el `LocaleSwitcher` (su `window.location` para el switch es intencional) ni el modelo URL-manda.

**Sitios corregidos** (superficie producto):

- Botón "Volver" de los perfiles (`PerfilClient.jsx`, `PerfilModel.jsx`): `history.goBack()` → `history.push('/client')` / `history.push('/model')` (dashboard del rol, determinista).
- Logos de cabecera de los perfiles: `<a href="/">` → `onClick` con `preventDefault` + `history.push('/')`.
- `Footer.jsx` (desktop y móvil): anchors internos (FAQ, Safety, Rules, Cookie Settings, marca) → `<Link>`.
- Botones "atrás" de las páginas legales/info (`Faq.jsx`, `Safety.jsx`, `Rules.jsx`, `Config.jsx`): `history.goBack()` → `history.push('/')`.

**Excepción documentada**: `/legal` es es-only por diseño (App.jsx redirige `/en/legal` → `/legal`). Su enlace en el footer se deja a propósito como anchor absoluto a `/legal`, con comentario en código para que no se "corrija" a `<Link>` por error (bajo `/en` generaría `/en/legal` y forzaría esa redirección con recarga).

**Reconocimiento**: este caso no se detectó al cerrar el frente del modelo URL-manda (commit `696eb6d`); se completa ahora. Desplegado y verificado a nivel de disponibilidad en AUDIT. Deuda de guardarraíl anti-reincidencia abierta en `known-debt.md`.

## 2026-05-31 — Arreglo del selector de idioma bloqueado tras login (producto) desplegado en AUDIT

Cierre del frente del bloqueo del selector de idioma ES/EN estando autenticado en la superficie producto: un usuario logueado (cliente o modelo) no podía cambiar el idioma de la UI; el cambio "se disparaba y volvía" al idioma por defecto. Sin sesión (público) y en admin funcionaba. Era una **regresión**.

**Causa raíz**: conflicto entre dos subsistemas de i18n que convivían sin chocar hasta que uno cambió. Por un lado, la URL como fuente de verdad estricta del locale mostrado (ADR-022: basename `/en` o su ausencia, resuelto en `App.jsx` e `i18n/index.js`). Por otro, `SessionProvider.applyLocale`, que reimponía `user.ui_locale` (preferencia persistida en BD) con `i18n.changeLanguage` en **cada carga autenticada**. Mientras el `LocaleSwitcher` de producto persistía la elección en BD, ambos coincidían. El commit `7520029` (16 mayo, "Paquete 1 rediseño CMS bilingüe") cambió el switcher de producto a navegación por URL **sin persistir** en BD; desde entonces la BD se quedaba en el valor de registro y `applyLocale` revertía el display tras el reload. El selector de admin (rama `updateUiLocale`) y el público no se vieron afectados.

**Diseño aplicado** (dos ficheros: `SessionProvider.jsx` + `LocaleSwitcher.jsx`): en producto/blog la **URL manda en lo que se muestra** — `applyLocale` deja de tocar `i18n` salvo en superficie admin (donde la URL no transporta locale y la preferencia sigue gobernando el render, sin cambios). La **elección se sigue persistiendo en BD**: el switcher de producto ahora hace `PUT /users/me/ui-locale` (solo con sesión, esperando el PUT antes de navegar) para que los emails transaccionales (`EmailLocaleResolver` → `EmailCopyRenderer`: bienvenida, baja, reset, verificación, rechazo de assets) y demás consumidores backend reciban el idioma correcto. La URL decide el display; la elección se guarda para emails y para recordar la preferencia.

**Patrón de backup de frontend estrenado en el despliegue**: el script `ops/scripts/deploy-frontend.ps1` hace `aws s3 sync --delete` (irreversible) y los buckets frontend de AUDIT no tienen versionado, sin rollback de frontend documentado. Se estrenó una red de rollback: antes de cada deploy se sincroniza el bundle vivo a un prefijo dedicado en el bucket de backups general — `s3://sharemechat-backups/audit/frontend/product/` y `.../admin/` — sobrescribible (siempre exactamente un respaldo, el del estado inmediatamente anterior), con red extra heredada del versionado del propio bucket de backups (el estado previo queda como versión noncurrent ~30 d por lifecycle). Procedimiento de rollback concreto por surface registrado en `docs/04-operations/runbooks.md`.

**Limitación conocida (decisión consciente, no bug)**: como la URL manda en el display de producto, un usuario logueado que entra a una URL **sin** prefijo `/en` verá español aunque su preferencia guardada sea inglés. Los **emails sí salen en el idioma correcto** (leen `user.ui_locale` de BD). Cerrar este matiz (p. ej. redirigir a `/en` en la entrada inicial según preferencia) sería una feature aparte y entraría en tensión con "no revertir una elección explícita", porque una URL de producto sin prefijo no permite distinguir "eligió español" de "entró por defecto". Queda fuera de scope.

## 2026-05-29 — Incidente acceso PSP Segpay: diagnóstico, fix ORP CloudFront y desactivación de country access en no-prod

Patricia (Segpay, PSP en onboarding) reportó dos síntomas en las superficies de AUDIT: no poder iniciar sesión (producto) y ver el overlay de mantenimiento (admin). El diagnóstico read-only (logs nginx + journal backend + estructura CloudFront + BD) reveló un **bug colateral del country access redesign del 2026-05-27**: las distribuciones admin de AUDIT y TEST usaban un Origin Request Policy (`admin-api-origin-request-v2` / `f11445e9`) que NO forwardea `CloudFront-Viewer-Country` al backend. c) **Las distribuciones CloudFront pueden divergir entre admin y producto con consecuencias graves.** El admin usaba un ORP restrictivo distinto al de producto, sin validación explícita de la diferencia. ## 2026-05-27 — Country access: de blocklist random a dual allowlist por flujo con bypass por IP

El country gate del backend pasa de una blocklist de 10 países sin criterio de negocio (`NL,JP,RO,PL,US,MX,CA,CH,NO,SG`, una prueba que nunca se revisó) a un modelo de allowlist por flujo siguiendo el patrón industrial deny-all + allow-explicit. Dos listas separadas: registro de cliente (28 países, criterio de poder adquisitivo) y registro de modelo (51 países, superconjunto que incluye LATAM hispanohablante completa, Brasil y Europa del Este por su oferta de modelos profesionales). Login, refresh y admin login usan la unión de ambas, de modo que una modelo registrada desde Colombia pueda loguearse aunque CO no esté en la lista de cliente. Se añade un bypass por IP (CSV de IPs/CIDR en `config.env`, resuelto con `IpAddressMatcher` de Spring Security) que salta toda validación de país; inicialmente solo la IP del operador, con las IPs de PSPs (CCBill y otros) pendientes de coordinación. El detonante fue el incidente de PSPs reportando falta de acceso, atribuible a la blocklist random que bloqueaba países desde los que operan.

La respuesta HTTP de bloqueo se unifica a un 403 genérico (`code: REGISTRATION_UNAVAILABLE`, mensaje "Registro no disponible", sin path ni scope ni país) para no filtrar a un atacante qué allowlist falló ni qué país fue rechazado; la razón real se registra solo en logs server-side. Se descartó alinear esta respuesta con la del Product Operational Mode (HTTP 503): la divergencia 403 (geo-restricción permanente) vs 503 (modo operativo temporal) es semánticamente correcta y útil para el cliente legítimo, y dentro del universo de bloqueos por país todas las respuestas ya son indistinguibles entre sí. Validado en TEST y AUDIT con login desde países permitido/no permitido y verificación del bypass; PROD queda pendiente del próximo arranque general.

## 2026-05-27 — Cierre del bypass de aislamiento auth producto→backoffice

Se cierra una regresión de seguridad descubierta durante la validación post-PRO-5.A: el endpoint público `POST /api/auth/login` no inspeccionaba el rol del usuario antes de emitir la cookie de sesión, y como la cookie lleva dominio con punto inicial (`.sharemechat.com`), era válida automáticamente en `admin.*`. Un usuario de backoffice podía autenticarse desde la superficie de producto y el frontend lo redirigía proactivamente al dashboard admin. El fix (Opción C) introduce una validación de doble dimensión en `AuthController.login` y `AuthController.refresh`: el rol primario (`users.role`) debe estar en `{USER, CLIENT, MODEL}`, y además el perfil backoffice cargado vía `BackofficeAccessService.loadProfile` debe estar vacío. Ambos checks devuelven `401 "Credenciales inválidas"` idéntico a credencial errónea, sin filtrar oráculo. `AdminAuthController.login` queda intacto como único endpoint válido para autenticar backoffice.

El aprendizaje central es que el modelo de roles tiene dos dimensiones independientes y el primer intento de fix (whitelist solo por `users.role`) era insuficiente: no cubría el caso real más común, un usuario con `users.role=USER` y rol backoffice asignado en la tabla `user_backoffice_roles` (por ejemplo SUPPORT), que pasaba el filtro de columna primaria y seguía accediendo al dashboard. Validar solo la columna de rol primario dejaba el agujero abierto; hizo falta consultar también el perfil backoffice efectivo. Tras el despliegue se purgó `refresh_tokens` en TEST y AUDIT para invalidar cualquier cookie emitida antes del fix. PROD pendiente del próximo arranque general; el bug no se expone mientras PROD arranque en modo PRELAUNCH, que ya bloquea el login de producto antes del controller.

## 2026-05-26 — Refactor config/secrets split en los tres entornos y aprendizaje sobre transcripción de secretos

Durante PRO-9 se separó el fichero `.env` monolítico de cada backend en dos: `config.env` (modo 0644, configuración no sensible) y `secrets.env` (modo 0600, credenciales), cargados en orden por systemd. El detonante fue un incidente: al inspeccionar la configuración del reporter perimetral de AUDIT, el agente ejecutó un `cat` que volcó el valor literal de `SMTP_PASSWORD` al chat. Es el segundo incidente de este tipo (el primero fue `DB_PASSWORD` el 2026-05-25, también transcrito y rotado). Ambas credenciales se rotaron de inmediato. La separación config/secrets es la mitigación estructural: reduce la superficie de exposición accidental y permite permisos más estrictos sobre el fichero de credenciales. En TEST el `secrets.env` quedó con propiedad `ec2-user` en lugar de `root` por su arranque manual sin systemd, una asimetría consciente documentada.

El aprendizaje permanente, aplicable a todos los frentes futuros con manejo de secretos, es doble: nunca hacer `cat` directo de un fichero que mezcla configuración y credenciales (filtrar preventivamente las claves de password con `grep -v` antes de mostrar el contenido), y tratar cualquier secreto que aparezca en un output como comprometido y forzar su rotación. La regla de no transcripción, antes implícita, queda reforzada como invariante operativo registrado en known-debt.md tras los dos incidentes consecutivos.

## 2026-05-23 — Reparación del test-access-blocker.service failed (paquete 10.B.3, cierre frente 10.B)

Aplicable a otros pipelines del proyecto donde haya dependencias entre fases con timers separados. ## 2026-05-23 — Allowlist de IPs operativas en pipeline perimetral (paquete 10.B.4)

Segundo sub-paquete del frente 10.B, cerrado el mismo día que el 10.B.1. Resuelve un riesgo de self-block detectado tras el primer email perimetral real enviado por TEST: la IP del administrador del proyecto (`90.175.201.51`) había sido clasificada como CRITICA con score 105 por su actividad de validación manual del refactor brief-per-locale (680 requests en AUDIT, 60 rutas distintas, ratios anómalos de 401/404 al onboarding del editor admin). En TEST el blocker está en DRY-RUN y no provoca daño, pero en AUDIT (Carril A real desde 2026-04-24) la siguiente validación intensiva habría escrito la IP del operador a `/etc/nginx/deny-audit-ips.conf` y le habría dejado fuera de su propio entorno. La solución es una allowlist por IP literal (variable `ALLOWLIST_IPS` del `config.env` del classifier) con short-circuit: cualquier IP listada salta toda la cadena de scoring y se clasifica como NORMAL/score=0/main_reason=allowlisted_ip. Aplicado a TEST y AUDIT con la única IP del operador (`90.175.201.51`). Validación funcional confirma que la IP ahora aparece como NORMAL en ambos entornos pese a tener requests >200 y distinct_routes >50; el `features` completo se conserva en el `summary.jsonl` para auditoría retrospectiva.

Aprendizaje operativo: el soporte de allowlist ya estaba **90% implementado** en el código del classifier (parseo CLI, `FeatureSet.allowlisted`, función dedicada `apply_allowlist_rule`), pero la lógica del descuento (-30 puntos) era cosmética: una IP con score natural ≥60 nunca caía por debajo de MALICIOSA. La elección entre escalar el valor (-200) o reescribir como short-circuit booleano se inclinó hacia lo segundo: garantiza el comportamiento independientemente del score natural y no requiere recalibrar si en el futuro se añaden reglas con weights mayores. Patrón aplicable: cuando un mecanismo de protección tiene magnitud insuficiente, evaluar si conviene reescribir como decisión booleana antes de la cadena en lugar de subir parámetros. El caso también recuerda otro patrón: cuando se activa un sistema de detección automática, la primera ejecución productiva debe revisarse manualmente para detectar falsos positivos sobre actividad legítima propia, antes de confiar en su output. Para PROD futuro, cuando el operador trabaje desde IPs variables (móvil, VPN, cafetería), la allowlist por IP no escalará y conviene complementarla con un mecanismo de header HTTP secreto que el normalizer reconozca antes de pasar las líneas al classifier; queda como sub-paquete posterior.

## 2026-05-21 — Cierre paquete 10.A.3.pre: red de seguridad de mantenimiento (SPA overlay activa + bucket S3 preparado)

Pre-requisito de 10.A.3 para que la ventana de parada del backend AUDIT (Flyway + V2 + JAR nuevo) no muestre errores feos al usuario. El plan original pedía Origin Failover de CloudFront como solución industrial automática. La realidad de la topología obligó a redibujar el diseño: el `DefaultCacheBehavior` de las distribuciones TEST y AUDIT sirve la SPA desde S3 directamente, no toca el backend, por lo que un OriginGroup ahí no detectaría backend caído. Solo los behaviors `/api/*`, `/match*`, `/messages*`, `/uploads/*`, `/assets/*`, `/sitemap.xml` y `/robots.txt` van al backend. Se aplican dos capas: capa SPA (activa) con interceptor que detecta 502/503/504 o HTML donde se esperaba JSON y muestra un overlay full-screen bloqueante hardcoded bilingüe (poll cada 30s a `/api/users/me` para detectar recuperación); capa CloudFront/S3 (preparada, no asignada) con bucket nuevo `sharemechat-maintenance` multi-env, OAC dedicado y bucket policy autorizando TEST + AUDIT, lista para un diseño futuro más completo.

Dos limitaciones de CloudFront aprendidas en este paquete justifican que el OriginGroup quede sin asignar. Primera: OriginGroup solo admite behaviors con `AllowedMethods` ⊆ `{GET, HEAD, OPTIONS}`, lo que excluye `/api/*` y `/uploads/*` (que son justamente los que la SPA usa para detectar el estado del backend). Segunda: S3 con OAC devuelve 403 (no 404) para objetos inexistentes, por lo que el bucket maintenance no puede servir el mismo `index.html` para cualquier path sin pre-popular cada path o sin una CloudFront Function de reescritura. Se aplicó OriginGroup a `/assets/*` + `/sitemap.xml` + `/robots.txt` como prueba, se observó el 403 del bucket cuando el backend estaba caído, y se revirtió CloudFront TEST al estado pre-cambio. La SPA con interceptor cubre el caso de uso real (usuario carga `/`, la SPA arranca, al primer fetch API recibe 5xx/HTML, overlay aparece, bloquea toda interacción, poll detecta recuperación cuando el backend vuelve). La validación visual del operador queda diferida a 10.A.3, donde el backend AUDIT se parará intencionalmente y la ventana es natural para confirmar el overlay en vivo.

## 2026-05-20 — Cierre del paquete 8 del rediseño bilingüe: split builder/validator y política de pintado del JSON

En un run productivo, el JSON emitido por `cms-json-builder` traía una comilla doble ASCII sin escapar dentro de `draft_markdown`; el gate cliente-side del CMS admin (paquete 6.7) lo detectó. Decisión: separar responsabilidades en dos skills (`cms-json-builder` solo construye; `cms-json-validator` valida y corrige sintaxis, fase 5.5 obligatoria del pipeline), con el orquestador encadenando ambas. Política de entrega final: el orquestador pinta el JSON validado en chat dentro de un bloque ` ```json ` en lugar de obligar al operador a abrir ficheros del sandbox; el sandbox de Cowork no escribe a `Downloads` de forma fiable y el gate cliente-side del admin ya bloquea cualquier JSON malformado al pegarlo.

Alternativas descartadas. Reforzar más la regla 15 del builder: una skill no puede auto-auditarse fiablemente porque el agente decide cuánto esfuerzo gasta en el self-check; el problema reaparecería. Validar exclusivamente en el backend sin tocar Cowork: pierde la corrección automática y deja toda la fricción del fallo en el operador, que tendría que diagnosticar y corregir el JSON a mano. Persistir `final.json` a `Downloads` como vía principal de entrega al operador: el sandbox no tiene permiso de escritura fiable a esa ruta, y mantener la regla obligatoria habría dejado deuda operativa permanente.

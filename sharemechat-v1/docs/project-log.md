# Bitácora del proyecto SharemeChat

Registro cronológico inverso (más reciente arriba) de hitos relevantes del proyecto: decisiones estructurales cerradas, cierres de fase o paquete, hipótesis archivadas, cambios de rumbo y aprendizajes operativos significativos. Cada entrada captura **qué pasó, por qué y, cuando aplica, qué se descartó y por qué**.

Este fichero **lo mantiene el agente local** (Claude Code u otro agente con acceso al repo) cada vez que materializa un cambio que cae en las categorías definidas. NUNCA se edita a mano. NO se rellena retroactivamente. NO se reescriben entradas pasadas.

La política operativa completa (categorías que disparan entrada, formato fijo, reglas de redacción y de inviolabilidad) vive en `docs/documentation-governance.md`, sección **Política operativa de la bitácora**.

## 2026-05-22 — Paquete 10.A.5: refactor URLs hardcoded a properties

Inspección de 10.A.4 (sync de assets) descubrió que `ModelContractManifestService.java` tenía la URL del manifest legal hardcoded en código Java a `https://assets.test.sharemechat.com/...`. AUDIT (y eventualmente PROD) hacían cross-environment fetch al bucket TEST en el flujo de aceptación del contrato de modelo. Se pausó el sync de assets y se abrió este paquete para refactorizar todas las URLs hardcoded en una sola pasada. El grep exhaustivo identificó 10 puntos: 5 en backend Java (`ModelContractManifestService`, `WebSocketConfig`, `SecurityConfig`, `ContentArticleService`, `SitemapController`), 3 cosméticos backend (`EmailVerificationService`, `PasswordResetService` con defaults `@Value` a TEST), y 2 frontend (`utils/runtimeSurface.js` con `ADMIN_APP_ORIGIN`/`PRODUCT_APP_ORIGIN` hardcoded; `styles/public-styles/HomeStyles.js` con URLs del CDN hardcoded en CSS-in-JS). Más placeholders i18n del CMS admin.

Refactor aplicado: 5 properties nuevas en `application.properties` (`app.assets.base-url`, `app.frontend.{product,admin}-origin`, `app.cors.allowed-origins`, `app.websocket.allowed-origins`) con defaults TEST, override de 3 de ellas en `application-audit.properties`; `ModelContractManifestService` reescrito para construir URLs con `@Value("${app.assets.base-url}")`; `WebSocketConfig` y `SecurityConfig` leen las listas como CSV → `String[]`; los dos fallbacks hardcoded de `ContentArticleService` y `SitemapController` reemplazados por `throw IllegalStateException` (sin fallback silencioso); defaults `@Value` de `EmailVerificationService` y `PasswordResetService` eliminados (la property DEBE estar definida o Spring falla el bootstrap). En frontend: nuevo `config/runtimeEnv.js` con `ENV_MAP` por hostname que resuelve `PRODUCT_ORIGIN`, `ADMIN_ORIGIN` y `ASSETS_BASE` en runtime; `runtimeSurface.js` y `HomeStyles.js` consumen del helper; placeholders i18n cambiados a `https://assets.sharemechat.com/<slug>.webp` (dominio neutro). Build local: `./mvnw clean package -DskipTests` BUILD SUCCESS en 22s, JAR 106 MB. `npm run build:product` exit 0, bundle delta ±10 B. JAR y bundle NO desplegados; quedan listos para deploy en próxima sesión operativa. Sub-paquete 10.A.4 (sync de assets) sigue pausado y se reanuda tras el deploy del refactor.

## 2026-05-21 — Cierre paquete 10.A.3.pre: red de seguridad de mantenimiento (SPA overlay activa + bucket S3 preparado)

Pre-requisito de 10.A.3 para que la ventana de parada del backend AUDIT (Flyway + V2 + JAR nuevo) no muestre errores feos al usuario. El plan original pedía Origin Failover de CloudFront como solución industrial automática. La realidad de la topología obligó a redibujar el diseño: el `DefaultCacheBehavior` de las distribuciones TEST y AUDIT sirve la SPA desde S3 directamente, no toca el backend, por lo que un OriginGroup ahí no detectaría backend caído. Solo los behaviors `/api/*`, `/match*`, `/messages*`, `/uploads/*`, `/assets/*`, `/sitemap.xml` y `/robots.txt` van al backend. Se aplican dos capas: capa SPA (activa) con interceptor que detecta 502/503/504 o HTML donde se esperaba JSON y muestra un overlay full-screen bloqueante hardcoded bilingüe (poll cada 30s a `/api/users/me` para detectar recuperación); capa CloudFront/S3 (preparada, no asignada) con bucket nuevo `sharemechat-maintenance` multi-env, OAC dedicado y bucket policy autorizando TEST + AUDIT, lista para un diseño futuro más completo.

Dos limitaciones de CloudFront aprendidas en este paquete justifican que el OriginGroup quede sin asignar. Primera: OriginGroup solo admite behaviors con `AllowedMethods` ⊆ `{GET, HEAD, OPTIONS}`, lo que excluye `/api/*` y `/uploads/*` (que son justamente los que la SPA usa para detectar el estado del backend). Segunda: S3 con OAC devuelve 403 (no 404) para objetos inexistentes, por lo que el bucket maintenance no puede servir el mismo `index.html` para cualquier path sin pre-popular cada path o sin una CloudFront Function de reescritura. Se aplicó OriginGroup a `/assets/*` + `/sitemap.xml` + `/robots.txt` como prueba, se observó el 403 del bucket cuando el backend estaba caído, y se revirtió CloudFront TEST al estado pre-cambio. La SPA con interceptor cubre el caso de uso real (usuario carga `/`, la SPA arranca, al primer fetch API recibe 5xx/HTML, overlay aparece, bloquea toda interacción, poll detecta recuperación cuando el backend vuelve). La validación visual del operador queda diferida a 10.A.3, donde el backend AUDIT se parará intencionalmente y la ventana es natural para confirmar el overlay en vivo.

## 2026-05-21 — Cierre paquete 10.A.2: fix CloudFront AUDIT (acme-challenge + CustomErrorResponses)

Tercer paso del frente 10.A (nivelación AUDIT a TEST). Un único `aws cloudfront update-distribution` sobre `E1ILXV7P6ENUV8` aplica los dos fixes simultáneos caracterizados en el pre-flight 10.A.1: cache policy del behavior `/.well-known/acme-challenge/*` pasa de `Managed-CachingOptimized` a `Managed-CachingDisabled` (replicación pura del fix de TEST en 10.A.0), y `CustomErrorResponses.Quantity` pasa de 2 a 0 eliminando las entradas `403 → /index.html` y `404 → /index.html` (replicación pura del hotfix de TEST del 2026-05-17). Precondición confirmada antes del cambio: la Function `redirect-spa-audit` ya estaba asociada al `viewer-request` del `DefaultCacheBehavior`, así que el SPA-fallback queda gestionado por la Function (no por `CustomErrorResponses`), exactamente como predecía la lección operativa del hotfix TEST. Propagación a `Deployed` en 32 s. Validación post-cambio confirma los dos fixes con `curl`: `/.well-known/acme-challenge/probe-<rand>` devuelve ahora `404` limpio desde nginx (antes devolvía `200` con `index.html` desde S3, enmascaramiento de `CustomErrorResponses`), y `/ruta-inexistente` sigue devolviendo `200` con `index.html` vía `redirect-spa-audit` Function (SPA-fallback intacto). Dos entradas de `known-debt.md` quedan formalmente cerradas: la del 2026-05-09 sobre cache policy acme-challenge (que ya se cerró parcialmente en 10.A.0 para TEST y mencionaba "validar también en AUDIT al hacer la nivelación", ahora cubierta) y la del 2026-05-17 sobre `CustomErrorResponses` AUDIT (marcada `[CERRADA]` con detalle de validación).

Hallazgo no anticipado: el bug 2 (`CustomErrorResponses` enmascarando 404) **ya se manifestaba en AUDIT**, no era latente. Una renovación HTTP-01 a través de CloudFront para `audit.sharemechat.com` habría fallado silenciosamente porque Let's Encrypt habría recibido `HTTP 200 + index.html` en lugar del token. No bloqueaba operación real porque `audit.sharemechat.com` se sirve con ACM (validación DNS automática) y `api.audit.sharemechat.com` se renueva por nginx directo sin pasar por CloudFront, pero el camino HTTP-01 vía edge quedaba roto. Lección operativa: cuando una deuda se cataloga "no urgente porque no se manifiesta todavía", verificar la asunción antes del fix; en este caso la asunción era cierta para `/api/*` con path variable pero falsa para `/.well-known/acme-challenge/`.

## 2026-05-21 — Cierre paquete 10.A.1: pre-flight AUDIT

Segundo paso del frente 10.A (nivelación AUDIT a TEST). Antes de tocar nada operativo en AUDIT, se prepara la red de seguridad y se inventaría el estado real del entorno. Se añade el alias SSH `audit-backend` al equipo del operador y se rellena el bloque `audit:` del mapping local con datos reales (3 distribuciones CloudFront, 6 buckets S3, endpoint RDS, schema, Function `redirect-spa-audit` ya asociada). Se crea el bucket multi-env `sharemechat-backups` (eu-central-1, privado, SSE-S3, versionado, lifecycle 90d) y se sube el `mysqldump` completo de la BD AUDIT a `s3://sharemechat-backups/audit/`. Se crea el bucket nuevo `sharemechat-content-private-audit` (réplica de la config del equivalente TEST) y el operador aplica desde la consola web la policy IAM inline `SharemechatContentPrivateAuditRW` al role `sharemechat-ec2-audit-role` (réplica literal de la policy análoga de TEST con los ARN cambiados). Smoke test desde la EC2 AUDIT confirma escritura/lectura/borrado sobre el bucket nuevo. Snapshot YAML del estado pre-cambio queda en `docs/_snapshots/state-audit-2026-05-21-1503.yaml`. La BD AUDIT confirma la pre-condición de ADR-025: 43 tablas, cero `content_*`, no existe `flyway_schema_history`. CloudFront AUDIT confirma dos bugs latentes (cache policy acme-challenge en `Managed-CachingOptimized` y `CustomErrorResponses 403/404 → /index.html` global) que se atajarán en el paquete 10.A.2.

Aprendizaje operativo: el `.env` de AUDIT usa comillas dobles en los valores; systemd las recorta automáticamente vía `EnvironmentFile`, pero un acceso manual desde shell debe sourcear el fichero (`set -a; source /opt/sharemechat/.env; set +a`) en lugar de hacer `grep | cut`, o las comillas se quedan dentro del valor y la autenticación a RDS falla con `Access denied`. También se confirma que el JAR en AUDIT vive en `/home/ec2-user/sharemechat-v1/` y no en `/opt/sharemechat/` como asumía el plan original 10.A.plan; el despliegue del JAR nuevo en 10.A.3 debe usar la ruta real.

## 2026-05-20 — Cierre paquete 10.A.0: cache policy acme-challenge corregida en CloudFront TEST

Primer paso del frente 10.A (nivelación AUDIT al estado actual de TEST). Antes de propagar nada a AUDIT, se cierra la deuda preexistente del 2026-05-09: el behavior `/.well-known/acme-challenge/*` en CloudFront TEST usaba `Managed-CachingOptimized` cuando debería usar `Managed-CachingDisabled`. La investigación previa confirmó que TEST tiene Certbot 2.6.0 con timer diario activo y dos certificados Let's Encrypt vivos (`test.sharemechat.com` en modo webroot vía `/usr/share/nginx/html`, `api.test.sharemechat.com` en modo nginx directo), próxima renovación efectiva ~2026-05-27. El cambio (TTL=0 puro, sin alterar routing ni headers) se aplica antes de esa ventana para dejar el patrón canónico activo y eliminar el riesgo acotado de que un 404 transitorio o un 5xx en una renovación quede cacheado. El bug latente análogo en AUDIT (`E1ILXV7P6ENUV8`) se atajará al inventariar AUDIT en el paquete 10.A.1.

Estrategia descartada: aplicar la corrección durante la nivelación de AUDIT en una sola ventana. Se rechazó porque mezclar fix en TEST con migración de AUDIT amplía el blast radius innecesariamente y porque cerrar TEST primero garantiza que el patrón nivelado en AUDIT sea ya el correcto (no se arrastra deuda).

Mejor anotarlas explícitamente en `known-debt.md` que arrastrarlas implícitamente entre sesiones.

## 2026-05-20 — Cierre del paquete 8 del rediseño bilingüe: split builder/validator y política de pintado del JSON

En un run productivo, el JSON emitido por `cms-json-builder` traía una comilla doble ASCII sin escapar dentro de `draft_markdown`; el gate cliente-side del CMS admin (paquete 6.7) lo detectó. La regla 15 del builder (self-check `JSON.parse` con hasta 3 intentos, introducida en 6.7) había sido saltada por el agente de Cowork sin que el operador pudiera detectarlo. Decisión: separar responsabilidades en dos skills (`cms-json-builder` solo construye; `cms-json-validator` valida y corrige sintaxis, fase 5.5 obligatoria del pipeline), con el orquestador encadenando ambas. Política de entrega final: el orquestador pinta el JSON validado en chat dentro de un bloque ` ```json ` en lugar de obligar al operador a abrir ficheros del sandbox; el sandbox de Cowork no escribe a `Downloads` de forma fiable y el gate cliente-side del admin ya bloquea cualquier JSON malformado al pegarlo.

Alternativas descartadas. Reforzar más la regla 15 del builder: una skill no puede auto-auditarse fiablemente porque el agente decide cuánto esfuerzo gasta en el self-check; el problema reaparecería. Validar exclusivamente en el backend sin tocar Cowork: pierde la corrección automática y deja toda la fricción del fallo en el operador, que tendría que diagnosticar y corregir el JSON a mano. Persistir `final.json` a `Downloads` como vía principal de entrega al operador: el sandbox no tiene permiso de escritura fiable a esa ruta, y mantener la regla obligatoria habría dejado deuda operativa permanente.
